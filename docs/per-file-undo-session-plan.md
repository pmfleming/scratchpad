# Per-File Undo Session Plan

The idea fits the current direction well: Scratchpad already has per-document operation undo in `TextDocument`, a newer `TextHistoryLedger` for global text history, and a piece-tree-backed edit engine. The strongest version is one ledger per open file, with each ledger keyed by buffer/file identity and carrying timestamps so a global history can be composed by merging open-file ledgers. Closing a file then has a clean rule: remove that file's ledger and all of its history entries from every visible history view.

The main pro is simplicity for normal editing. Per-file undo becomes the natural local case, global undo becomes an ordered merge of per-file entries, and users can move backward or forward to meaningful points in time for just one file without disturbing unrelated open files. The piece tree can provide stable text operation payloads without falling back to broad workspace snapshots, which also aligns with the project goal of keeping undo focused on text changes rather than file, tab, layout, or settings actions.

The main con is drift, but the project already has a useful decision point: when the underlying file has changed, Scratchpad can ask whether to use the on-disk version or the session version. Choosing the on-disk version should necessarily discard that file's ledger, because the history no longer describes the restored buffer. Choosing the session version can keep the ledger, but time-travel still needs validation because later edits can make an older point impossible to replay back to an identical appearance.

Plan: first make each open file's in-memory ledger the source of truth for that file's text undo, with entries carrying timestamp, source label, undo/redo operation payload, selection restore metadata, and a revision fingerprint. Then build the global history as a merged view across currently open ledgers, with file close pruning that simply drops the closed file's ledger. Finally, connect session restore to the existing disk-versus-session choice: on-disk restore drops that file's ledger, session restore may keep it if fingerprints validate, and unsafe history should be discarded with a clear warning.

An embedded piece-tree variant is also plausible. Each inserted or loaded span could carry cheap provenance metadata such as timestamp and source (`load`, `edit`, `paste`, `search_replace`), while cut/deleted spans could be retained as invalid/deleted history records rather than physically disappearing from the history model. The existing file/session metadata can still decide whether a restored tree is valid: if the user chooses the on-disk file after an external change, the embedded history for that file is discarded; if the session version is kept, the tree history may remain usable. The tradeoff is that per-file history becomes very natural, but global history becomes a derived merge across multiple tree histories and will need careful ordering, pruning, and UI rules.
