# Button Specification

This document defines the button system for the custom window chrome and tab strip used by this project.

The intent is to give a stable spec that can be used to create or refactor Rust source files without re-deciding button names, layout rules, event responsibilities, or state behavior.

## Scope

This spec covers:

- Header action buttons
- Window caption buttons
- Per-tab buttons in the custom tab strip
- Text labels and state behavior
- Layout constraints
- Event routing expectations
- Recommended Rust file boundaries for implementation

This spec does not cover:

- Visual styling beyond text labels and sizing
- Theme/color system
- Keyboard shortcut registration outside menu definitions
- Overflow scrolling for large tab counts

## Design Goals

- Keep the header fully custom and independent from the native title bar
- Preserve native editor behavior through `nwg::TabsContainer` and `nwg::Tab`
- Treat visible buttons as the user-facing source of interaction
- Keep each button's responsibility narrow and predictable
- Make it easy to move button construction and behavior into dedicated files later

## Button Inventory

There are three button groups.

### 1. Header Action Buttons

These are placed in the custom header, to the right of the visible tab strip and to the left of the window caption buttons.

#### Add Tab Button

- Internal field name: `new_tab_button`
- Kind: header action button
- Default label: `+`
- Purpose: create a new untitled tab and select it
- Parent control: `header`
- Primary event: `OnButtonClick`
- Primary action: call `EditorApp::new_tab()`
- Failure behavior: show modal error using `show_error("New Tab Failed", ...)`

#### Menu Button

- Internal field name: `menu_button`
- Kind: header action button
- Default label: `v`
- Purpose: open the file popup menu
- Parent control: `header`
- Primary event: `OnButtonClick`
- Primary action: call `EditorApp::popup_file_menu()`
- Failure behavior: none expected during normal popup display

### 2. Window Caption Buttons

These replace the native title bar buttons.

#### Minimize Button

- Internal field name: `minimize_button`
- Kind: caption button
- Default label: `_`
- Purpose: minimize the window
- Parent control: `header`
- Primary event: `OnButtonClick`
- Primary action: call `window.minimize()`

#### Maximize / Restore Button

- Internal field name: `maximize_button`
- Kind: caption button
- Default label when normal: `[ ]`
- Default label when maximized: `o`
- Purpose: toggle maximize/restore state
- Parent control: `header`
- Primary event: `OnButtonClick`
- Primary action: call `EditorApp::toggle_maximize()`
- State dependency: label must be refreshed through `sync_caption_buttons()` after maximize/restore or resize-related state changes

#### Close Window Button

- Internal field name: `close_window_button`
- Kind: caption button
- Default label: `X`
- Purpose: close the window
- Parent control: `header`
- Primary event: `OnButtonClick`
- Primary action: call `window.close()`
- Current behavior note: this uses the existing window close path and does not yet perform a full multi-tab dirty-state sweep

### 3. Custom Tab Strip Buttons

Each visible tab in the custom header is represented by two buttons.

#### Tab Select Button

- Internal field name in `TabPage`: `tab_button`
- Kind: dynamic per-tab button
- Parent control: `header`
- Purpose: select the tab represented by the page
- Primary event: `OnButtonClick`
- Primary action: call `EditorApp::select_tab(index)`
- Label source: generated by `custom_tab_title(...)`

Label rules:

- Unsaved tabs prepend `*`
- Active tab is wrapped in `[` and `]`
- Inactive tab uses plain label text
- Label uses file name if available, otherwise `Untitled {n}`

Examples:

- `notes.txt`
- `*notes.txt`
- `[notes.txt]`
- `[*notes.txt]`
- `Untitled 3`

#### Tab Close Button

- Internal field name in `TabPage`: `close_button`
- Kind: dynamic per-tab button
- Parent control: `header`
- Default label: `x`
- Purpose: close a specific tab
- Primary event: `OnButtonClick`
- Primary action: call `EditorApp::close_tab(index)`
- Close behavior: must respect unsaved-change confirmation rules

## Layout Rules

These constants are the current source of truth.

- `HEADER_HEIGHT = 38`
- `HEADER_PADDING = 6`
- `TAB_HEIGHT = 28`
- `TAB_GAP = 4`
- `TAB_CLOSE_WIDTH = 24`
- `TAB_SLOT_MAX_WIDTH = 190`
- `TAB_SLOT_MIN_WIDTH = 72`
- `ACTION_BUTTON_WIDTH = 32`
- `ACTION_BUTTON_HEIGHT = 28`
- `WINDOW_BUTTON_WIDTH = 42`
- `WINDOW_BUTTON_HEIGHT = 28`

### Horizontal Ordering

From left to right, the header layout is:

1. Dynamic tab buttons
2. Add tab button
3. Menu button
4. Minimize button
5. Maximize/restore button
6. Close window button

### Positioning Rules

- All header buttons use `HEADER_PADDING` as the top offset
- The right-side controls are laid out from right to left
- The tab region consumes remaining space to the left of the action buttons
- Each visible tab consumes one slot composed of:
  - `tab_button` width
  - `close_button` width
  - trailing `TAB_GAP`

### Tab Width Calculation

Tab width is derived from the available area:

`slot_width = clamp((available_width - total_gaps) / tab_count, TAB_SLOT_MIN_WIDTH, TAB_SLOT_MAX_WIDTH)`

Where:

- `available_width = right_edge - left_edge`
- `total_gaps = (tab_count - 1) * TAB_GAP`

### Overflow Behavior

Current behavior:

- Tabs that do not fit in the available area are hidden
- No scrolling or overflow dropdown currently exists

Future-compatible expectation:

- Overflow logic should be able to replace hiding behavior without changing button identifiers or responsibilities

## State Rules

### Dirty State

- Dirty state is derived from `page.editor.modified()`
- Dirty state affects:
  - window title
  - native tab title
  - custom tab label
- Dirty state must be refreshed after text input and after successful saves

### Active Tab State

- Active tab is derived from `tabs.selected_tab()`
- Active state affects:
  - focused editor control
  - window title
  - custom tab label formatting

### Planned Unified Activity And Selection

Goal:

- Collapse the current visual distinction between `active` and `selected` tab slots into one visible state.
- Active tabs and selected tabs should both render as a darkened tab with the same blue outlined square treatment.
- The active tab remains a distinct logical concept for focus, command routing, and window-title state, but it no longer has a separate visual style.

Proposed behavior:

- The active tab must always be included in the selected tab-slot set.
- A normal tab click should activate the clicked tab and reset selection to only that tab.
- `Ctrl` click should add or remove extra selected tabs, but must never leave the active tab outside the selected set.
- `Shift` click should extend selection from the current anchor while ensuring the active tab remains selected.
- Multi-selection should be treated as `selected tabs`, with exactly one of them also marked as the active tab for focus-sensitive actions.

Implementation plan:

- Replace the separate active-only paint path and selected-overlay paint path with one shared selected-state visual treatment.
- Keep `active_tab_index` for editor focus and command targeting, but define it as a guaranteed member of `selected_tab_slots`.
- Tighten selection mutations so any activate/select flow preserves the invariant `active in selected_tab_slots`.
- Audit overflow-list, drag/drop, context-menu, and settings-tab rendering so they all use the same unified selected appearance.
- Update any docs and tests that currently describe active and selected tabs as visually different states.

Acceptance notes:

- Users should no longer need to infer meaning from two tab highlight styles.
- Single-selection should still feel unchanged: one active tab that is also visibly selected.
- Multi-selection should read as `one active tab inside a selected group`, not as two competing highlight systems.

### Maximize State

- Maximize state is derived from `IsZoomed(hwnd)`
- The maximize button must update its label when the window changes between normal and maximized states

## Event Routing Rules

Buttons should not contain business logic themselves. Event handlers should route into `EditorApp` methods.

### Required Routing Map

- `new_tab_button` -> `new_tab()`
- `menu_button` -> `popup_file_menu()`
- `minimize_button` -> `window.minimize()`
- `maximize_button` -> `toggle_maximize()`
- `close_window_button` -> `window.close()`
- `tab_button` -> `select_tab(index)`
- `close_button` -> `close_tab(index)`

### Context Menu Rules

- Right-click on a tab button or close button should first select the corresponding tab
- After selection, show the tab popup menu via `popup_tab_menu()`

### Drag Behavior

- Left mouse down on unused header space should begin window drag
- Buttons themselves should remain clickable and must not initiate drag as their primary behavior

## Construction Rules

### Header Buttons

All static header buttons should be built during app construction in `EditorApp::build()` or in a function delegated from it.

Required builder pattern:

```rust
nwg::Button::builder()
    .text("+")
    .parent(&app.header)
    .build(&mut app.new_tab_button)?;
```
```

### Per-Tab Buttons

Each new page must create:

- one `nwg::Tab`
- one `nwg::TextBox`
- one `tab_button`
- one `close_button`

These four controls belong to one `TabPage` and should be created together inside tab creation flow.

## Suggested Rust File Layout

If button code is expanded further, use this structure.

### Minimum Viable Split

- `src/app/build.rs`
  - create all static header buttons
  - create menus and dialogs
- `src/app/tabs.rs`
  - create per-tab buttons
  - generate per-tab labels
  - delete per-tab buttons with their page
- `src/app/chrome.rs`
  - layout header buttons
  - update maximize label
  - start drag behavior
- `src/app/events.rs`
  - map button click events to `EditorApp` methods

### Optional Next Split

If you want a more explicit button subsystem, create these files:

- `src/app/buttons.rs`
  - shared button labels
  - helper constructors for static header buttons
  - helper constructors for dynamic tab buttons
- `src/app/button_ids.rs`
  - semantic identifiers or enum names for button roles
- `src/app/button_layout.rs`
  - size constants and placement helpers

## Suggested Data Model

If button state becomes denser, move away from raw fields on `EditorApp`.

Recommended structures:

```rust
pub(crate) struct HeaderButtons {
    pub(crate) new_tab: nwg::Button,
    pub(crate) menu: nwg::Button,
    pub(crate) minimize: nwg::Button,
    pub(crate) maximize: nwg::Button,
    pub(crate) close: nwg::Button,
}

pub(crate) struct TabButtons {
    pub(crate) select: nwg::Button,
    pub(crate) close: nwg::Button,
}
```
```

And then:

```rust
pub(crate) struct TabPage {
    pub(crate) tab: nwg::Tab,
    pub(crate) editor: nwg::TextBox,
    pub(crate) buttons: TabButtons,
    pub(crate) _layout: nwg::GridLayout,
    pub(crate) path: Option<PathBuf>,
    pub(crate) untitled_id: usize,
}
```
```

## Naming Rules

Use these naming conventions consistently:

- Static button field names end with `_button`
- Dynamic per-tab close/select controls use semantic names, not positional names
- Methods should express intent, not widget mechanics

Preferred:

- `new_tab_button`
- `close_window_button`
- `toggle_maximize()`
- `sync_caption_buttons()`
- `layout_tab_buttons()`

Avoid:

- `btn1`
- `header_btn`
- `do_tab_thing()`
- `update_buttons_misc()`

## Acceptance Criteria

A button implementation is compliant with this spec if:

- All header and tab buttons have a single, stable responsibility
- Button click handlers route into `EditorApp` methods
- Layout logic preserves the left-to-right header order defined above
- Dirty state is reflected in tab and window titles
- Maximize button label changes with maximize state
- Tab close buttons respect unsaved-change confirmation
- New tabs produce both editor content and visible custom tab buttons
- `cargo check` succeeds after any button-related file creation or refactor

## Current Gaps

These are known missing features, not spec violations.

- No hover/pressed/disabled visual state system
- No overflow scroller for many tabs
- No full app close confirmation across all dirty tabs
- No dedicated button helper module yet

## Implementation Guidance

When creating new source files from this spec:

- Keep widget construction close to ownership boundaries
- Keep event mapping centralized
- Do not duplicate label-generation logic in multiple files
- Keep Win32-specific chrome code isolated from file I/O and tab creation logic
- Prefer extracting small helper methods before introducing more structs

This should be treated as the source of truth for future button refactors unless the UI design changes.
